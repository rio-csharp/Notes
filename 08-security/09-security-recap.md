# Security Recap

Security is a set of trust boundaries. Tokens, OAuth flows, authorization rules, password handling, browser storage, injection defenses, and rate limiting all matter because they define who is trusted, what is exposed, how long credentials remain useful, and how the system behaves under abuse or compromise. Once those boundaries are made explicit, individual controls become easier to reason about in context.

Several themes connect the material. First, representation is not trust. A JWT payload, a frontend permission check, or a browser-held token becomes meaningful only after the server validates the right boundary. Second, security controls trade one kind of risk against another. Cookies shift pressure toward CSRF, browser-readable storage shifts pressure toward XSS, cached authorization improves speed but weakens freshness, and stronger abuse controls may challenge availability. Third, operational discipline matters as much as primitives. Rotation, revocation, auditability, structured logging, dependency hygiene, and rate limiting are what keep a theoretically sound design resilient in practice.

Taken together, these ideas move security away from isolated vulnerability facts and toward a more useful engineering model: identify the trust boundary, understand what an attacker could steal, replay, or misuse there, and then apply controls that fit that specific boundary. That model scales much better than memorizing scattered security slogans.
