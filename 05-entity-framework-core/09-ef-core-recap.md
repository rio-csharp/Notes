# EF Core Recap

This chapter presented EF Core as a persistence runtime rather than as a thin database helper. `DbContext` defines the unit-of-work boundary, the change tracker translates object state into relational commands, and query behavior depends on provider translation rather than on ordinary in-memory LINQ. Once those foundations are clear, the rest of EF Core becomes easier to reason about because relationships, migrations, transactions, converters, raw SQL, and performance tuning all turn out to be consequences of the same core execution model.

Several themes connect the chapter. First, persistence boundaries should remain explicit. Context lifetime, tracking mode, transaction scope, and migration rollout strategy all work best when they match real operational boundaries instead of being treated as defaults to ignore. Second, EF Core should be used in ways that respect the database rather than abstract it away. Query shape, indexes, schema evolution, locking, and provider-specific behavior still matter. Third, richer domain models are valuable only when they remain compatible with persistence clarity. Value objects, owned types, and custom mappings should strengthen the model without obscuring how the system actually stores and queries data.

Taken together, these chapters define a mature mental model for using EF Core professionally. The framework is powerful because it bridges object-oriented application code and relational persistence, but that bridge has costs and constraints. Teams that understand those costs tend to produce code that is easier to operate, easier to evolve, and less likely to drift into either naive ORM usage or premature SQL fragmentation.
