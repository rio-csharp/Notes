# C# Recap

C# is not only a syntax for expressing control flow. It is a design language for shaping types, defining contracts, preserving invariants, composing queries, coordinating asynchronous work, and controlling failure boundaries. The chapters in this part have traced that design language from its foundations through the engineering decisions that shape production systems.

## Type Design Decisions

The choice among `class`, `struct`, and `record` carries consequences far beyond the keyword itself. Reference types (`class`, `record`) participate in identity-based equality by default and live on the managed heap, where allocation cost and garbage collection pressure become relevant at scale. Value types (`struct`, `record struct`) use value-based equality and can live on the stack or inline within containing objects, avoiding heap allocations but introducing copying costs when passed around. Records — both reference and value — synthesize equality members from the type's shape, which makes them natural choices for immutable data carriers, value objects in domain models, and DTOs at API boundaries (Chapter 01, Chapter 02).

These decisions propagate into API contracts. A property returning a mutable reference type exposes shared state unless the type is immutable by construction. Returning a struct copies the value, preventing unintended shared mutation but potentially surprising callers when modifications do not propagate back. Sealing a class with `sealed` communicates finality and enables compiler optimizations including devirtualization. These are not cosmetic choices; they determine what incorrect states the type system permits and what guarantees callers can rely on at compile time (Chapter 02, Chapter 03).

Primary constructors, introduced in C# 12, further compress the relationship between data and type by binding constructor parameters directly to the type body, reducing the ceremony of field declaration and assignment for simple cases while preserving the option to expand into explicit constructors when initialization logic grows complex (Chapter 01).

## Encapsulation And Abstraction

Access modifiers form the first layer of encapsulation: `private` hides implementation details, `protected` opens extension points to derived classes, `internal` draws a boundary at the assembly, and `public` establishes the type's external contract. These are not merely syntax. They encode design intent about which parts of the system are allowed to change independently and which changes would ripple across assembly boundaries (Chapter 03).

Abstract classes and interfaces represent two distinct abstraction mechanisms. An abstract class carries implementation — shared fields, constructors, default method implementations — and establishes an "is-a" relationship that implies a common base. An interface defines a capability contract that unrelated types can fulfill, and since C# 8, interfaces can carry default implementations, narrowing the practical gap between the two constructs. The choice turns on whether the abstraction represents a shared partial implementation (abstract class) or a capability that crosses type hierarchies (interface). Composition over inheritance remains the prevailing design discipline: delegating behavior to collaborating objects rather than inheriting it produces types that are easier to test in isolation and less brittle under change (Chapter 03).

## Generics And Collections

Generics provide the mechanism for writing type-safe abstractions that the compiler verifies at usage sites. Generic constraints (`where T : class`, `where T : struct`, `where T : IComparable<T>`, `where T : new()`) narrow the type parameter to the capabilities the abstraction actually requires, trading flexibility for compile-time safety (Chapter 04). Covariance (`out`) and contravariance (`in`) on interface type parameters enable polymorphic assignment — `IEnumerable<string>` is an `IEnumerable<object>` — but only when the compiler can prove the variance is safe according to the position the type parameter occupies in the interface's member signatures (Chapter 04).

Collection choice connects generics to runtime behavior. `List<T>` provides contiguous storage indexed by position. `Dictionary<TKey, TValue>` offers O(1) lookup by hash code at the cost of higher memory overhead per entry. `HashSet<T>` enforces uniqueness without the value payload of a dictionary. `LinkedList<T>` enables O(1) insertion and removal at known nodes but eliminates random-access indexing. `ImmutableArray<T>` and `ImmutableList<T>` provide structural sharing under modification — every "mutation" produces a new collection while reusing most of the existing structure — making them suitable for scenarios where snapshots and thread safety matter more than write throughput (Chapter 05).

LINQ sits atop this collection infrastructure as a composable query pipeline. Deferred execution — the principle that a LINQ query is not evaluated until enumeration — means that queries composed from `Where`, `Select`, and `Take` do not traverse the source until a terminal operator forces materialization. `IQueryable<T>` extends this to expression trees: the query is captured as a data structure that a provider such as Entity Framework Core translates into SQL, postponing execution until the database round-trip. The distinction between `IEnumerable<T>` (in-memory) and `IQueryable<T>` (expression-tree) is not academic; it determines whether filtering happens on the application server or the database, with consequences for network transfer, memory pressure, and index utilization (Chapter 06).

## Asynchronous And Concurrent Execution

The async/await model separates asynchronous coordination from threading. An `async` method compiles into a state machine that captures local state, tracks completion status, and schedules continuations via the `SynchronizationContext` or `TaskScheduler` current at the point of the `await`. The method does not own a thread during the await; the thread returns to the pool, and the continuation runs on whatever thread the scheduler assigns. This is why `Task.Result` and `Task.Wait()` can deadlock in synchronization-context-aware environments: they block the thread that the continuation expects to run on (Chapter 07).

Concurrency introduces shared state, and shared state introduces synchronization. `lock` provides mutual exclusion on an object monitor. `SemaphoreSlim` limits concurrent access to a configurable count, useful for throttling. `ReaderWriterLockSlim` distinguishes read-heavy from write-heavy access patterns, allowing concurrent readers when no writer is active. `Channel<T>` from `System.Threading.Channels` provides an asynchronous producer-consumer queue with bounded capacity and backpressure — when the channel is full, writers either block asynchronously or drop items, depending on the configured `BoundedChannelFullMode`. This replaces many hand-rolled producer-consumer patterns and integrates naturally with `await foreach` (Chapter 08).

`System.Threading.Tasks.Dataflow` extends the concurrency toolkit further with blocks such as `ActionBlock<T>`, `TransformBlock<TInput, TOutput>`, and `BufferBlock<T>` that compose into processing pipelines with explicit parallelism and throttling. The choice between channels, dataflow blocks, and raw `Task.WhenAll` depends on whether the workload needs backpressure, ordered processing, fan-out/fan-in, or configurable degrees of parallelism (Chapter 08).

## Failure And Reliability

Exception handling across layers produces a system where failures carry semantic meaning rather than simply propagating raw stack traces. Domain exceptions protect invariants and stop invalid state transitions. Application services translate infrastructure exceptions into the application's own failure vocabulary. API and messaging boundaries convert exceptions into protocol-appropriate responses — HTTP problem details or dead-letter directives — keeping internal diagnostics rich while external contracts remain stable (Chapter 09).

Cancellation is distinct from failure. `OperationCanceledException` caught with a filter that checks `CancellationToken.IsCancellationRequested` distinguishes caller-initiated cancellation from unexpected timeouts, preventing noise in logs and metrics. Retries apply only to failures whose root cause is plausibly transient: network timeouts, temporary unavailability, throttling responses. Retrying a validation error, an authorization denial, or an invariant violation repeats a guaranteed failure. In message-driven systems, this distinction extends to the broker: some exceptions dead-letter immediately because the message will never succeed, while others abandon the message for redelivery with a finite delivery-count limit (Chapter 09).

The `ExceptionDispatchInfo` class preserves original stack traces when exceptions must be captured and rethrown across asynchronous boundaries. Exception filters execute before stack unwinding, enabling diagnostic enrichment and handling decisions at the richest possible point of visibility. Together, these mechanisms form a failure-handling surface that respects both diagnostic needs and operational reliability (Chapter 09).

## Choosing The Right Abstraction

The design philosophy that emerges from these chapters is not about language mastery for its own sake. It is about choosing abstractions that match the problem's shape: value types for lightweight data that travels with its owner, reference types for entities with identity, immutable records for snapshots and DTOs, generic constraints that express exactly the capabilities required, async patterns that free threads rather than occupying them, and exception types that name failure categories the surrounding system can reason about.

Compile-time feedback is preferred wherever the language can express the constraint. A generic constraint catches misuse at build time that a runtime cast would only surface in production. A sealed class prevents unintended extension before it happens. A nullable reference type annotation exposes nullability at the call site rather than in a `NullReferenceException` stack trace. These are not ideological preferences; they are practical reductions in the surface area of production failures.

Execution cost is part of the design calculus. Heap allocations, boxing, LINQ materialization timing, lock contention, and channel backpressure are not implementation details to be addressed after the fact. They are first-order consequences of type and algorithm choices that compound under load. Professional C# code is judged not by language cleverness but by whether the code expresses meaning clearly, chooses the right abstraction for the job, and preserves predictable behavior as systems grow in size, load, and complexity.
