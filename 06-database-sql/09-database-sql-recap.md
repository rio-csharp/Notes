# Database And SQL Recap

This chapter introduced relational databases as integrity systems rather than as passive storage. Tables, keys, and constraints define the basic structure, but the deeper value lies in how that structure preserves valid relationships and rejects invalid state even when application code is imperfect. SQL then operates over that structure declaratively, expressing result sets, filters, groupings, and transactional boundaries in terms the optimizer can translate into execution plans.

Several themes connect the chapter. First, data shape and query shape are inseparable. Joins, normalization, denormalization, indexes, and query plans all reflect the same underlying truth: the schema determines which facts live where, and the workload determines how expensive it is to retrieve or update them. Second, physical design matters. Indexes, cardinality estimates, lock scope, and access paths influence both performance and concurrency behavior. Third, advanced patterns such as denormalized read models, partitioning, or sharding should extend a sound core model rather than compensate for a weak one.

Taken together, these chapters provide the relational foundation for later application and architecture work. Once schema design, SQL semantics, access paths, and transaction behavior are understood at the database level, higher-level persistence tooling becomes easier to evaluate because its abstractions can be judged against the real behavior of the underlying system.
